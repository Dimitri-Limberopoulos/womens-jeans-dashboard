Cross-Retailer Product Dashboard  |  End-to-End Pipeline Guide

How to scrape product data from any set of retail websites, clean and classify it, and render an interactive single-file HTML dashboard — from zero to deployed. No server, no database, no API keys.

1. Collect Product URLs

Use Instant Data Scraper to grab every product URL from a retailer's category page. One CSV per retailer.

Setup

Install Instant Data Scraper from the Chrome Web Store. It's free and requires no configuration.

How It Works

1Navigate to the retailer's full category page (e.g., target.com/c/womens-jeans). Make sure the page title says "All" — not a filtered subset.
2Click the Instant Data Scraper icon. It auto-detects the product grid. If it picks the wrong table, click "Try another table" until the URLs appear.
3Click "Start crawling" — it paginates automatically through all pages.
4When finished, click Download → CSV. Save to your project folder as {retailer}.csv.
For retailers that use infinite scroll instead of numbered pages, scroll to the bottom manually first to load all products, then run the scraper. De-duplicate URLs before moving to Phase 2 — some retailers list the same product across multiple category views.

Output

One CSV per retailer. The scraper captures many columns — you only need the URL column (typically the first column, or one named "href" / "link").

2. Scrape Product Detail Pages

Visit each URL, extract 10–15 product attributes from the DOM. One scraper per retailer, with crash recovery built in.

Install Dependencies

pip3 install playwright openpyxl beautifulsoup4 --break-system-packages python3 -m playwright install chromium

Choose Your Scraping Method

Not every site needs a full browser. Sites that render product data server-side can use the much faster web_fetch approach. Sites with heavy JavaScript rendering or bot detection need Playwright.

MethodSpeedUse WhenExample Sites
web_fetch~1.5s/pageProduct data is in the initial HTML responseTarget, Old Navy, Macy's
Playwright~3–4s/pageData loads via JavaScript, or site has bot detectionWalmart, Amazon, Levi's

Building a Scraper for a New Site

1Inspect the target site. Open a product page in Chrome → right-click a data element (price, color swatch, product name) → Inspect. Note the CSS selector or data-* attribute.
2Test selectors in DevTools. Run document.querySelectorAll('YOUR_SELECTOR').length in the console. Verify it returns the right count.
3Write a single-URL test script. Scrape one known product and verify every field before running the full batch.
4Build the batch scraper with crash recovery — save results every 100 URLs so a crash doesn't lose hours of work.

Crash Recovery Pattern

Every scraper saves a {retailer}_pdp_progress.json checkpoint after each batch of 100 URLs. On restart, already-scraped URLs are skipped automatically. Just re-run the same command — it picks up where it left off.

Multi-Pass Strategy

Expect ~5–10% of URLs to fail on the first pass (timeouts, CAPTCHAs, 503 errors). Plan for it:

Pass 1 — Bulk Scrape
Run all URLs. Accept some "Unknown" or "ERROR" entries. Save results + error log.
Pass 2 — Error Recovery
Re-scrape only failed URLs with lower concurrency and longer timeouts.
Pass 3 — Unknown Resolution
For fields like color names that returned "Unknown", try alternative selectors or a different scraping method.
Pass 4 — Classification
Run regex classifiers on extracted text. Batch web-search any names the regex can't handle.
For long runs (1,000+ URLs, 2–4 hours): keep your Mac plugged in, enable "Prevent automatic sleeping" in System Settings → Battery, and don't close the terminal or browser window. You can use other apps freely while it runs.

Anti-Detection Basics

# Launch visible browser (harder to detect than headless) browser = await p.chromium.launch(headless=False, args=["--disable-blink-features=AutomationControlled"]) # Mask the webdriver flag await context.add_init_script( "Object.defineProperty(navigator, 'webdriver', {get: () => undefined})" ) # Use a real-looking user agent context = await browser.new_context( user_agent="Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7) ...", viewport={"width": 1280, "height": 800} )

Adaptive Throttling

Instead of fixed sleep times, back off when errors spike and speed up when things are stable. Start at 1.5s delay. On error, multiply by 1.5× (max 10s). After 5 consecutive errors, pause 30s. On success, gradually reduce by 5% (min 1.0s).

3. Clean, Classify, and Unify

Normalize all retailer data into one consistent format. Classify creative color names. Backfill missing fields.

Unify Retailer Data

Each scraper outputs slightly different column names and formats. A unification script normalizes everything into a single CSV — standardizing brand names, parsing prices (stripping "$" and commas), assigning retailer groups, and normalizing pack sizes for multi-packs.

python3 unify_retailers.py # Reads: all *_pdp_results.xlsx files # Writes: unified_retailer_data.csv

Color → Wash Classification

Raw color names from retailers are a mix of standard names ("Dark Wash") and creative brand-specific names ("Ink Well", "Honey Crisp", "Salsa Verde"). A two-stage pipeline handles this:

Stage 1 — Word-Boundary Regex (~85% coverage)
Patterns like \b(dark|dk\b|rinse|indigo)\b map most colors to standard categories: Light Wash, Medium Wash, Dark Wash, Black, White/Cream, Grey, Colored.

Word boundaries are critical. Without \b, "light" matches "moonlight" and "grey" matches "greyhound". This was the single biggest source of misclassification bugs.
Stage 2 — Web Search for Creative Names (~10–12% more)
Batch the remaining unclassified names into groups of 20. Search "What color is [Name1], [Name2], ... in jeans?" Parse results into a lookup table. This resolves most brand-specific names that regex can't handle.

Field Normalization

ChallengeSolution
Fit Style vs Leg Shape overlapWhen one field is missing, fall back to the other using a mapping table (e.g., "Straight" leg shape → "Straight" fit)
Walmart multi-packsDetect pack size from product name (regex for "2-pack", "3-pack"), divide price by pack count

4. Build the Dashboard

Assemble a self-contained HTML file with all data, CSS, and JavaScript embedded. No server required.

Architecture

The build uses three source files that get combined into one HTML file. This separation exists because embedding complex JavaScript directly in a Python f-string causes syntax errors from template literals and special characters.

app.js
All chart logic, filters, toggles, interactions. Must use ES5 syntax — no arrow functions, template literals, or spread operators.
data.json
Compact product data from Phase 3. Embedded as var RAW = [...]; in a script tag.
build.py
Reads both files, assembles the complete HTML with CSS theme, CDN links, and the side nav layout.
python3 build.py # Reads: app.js, data.json # Writes: index.html (~1.9 MB)

Why ES5?

The JavaScript is embedded via Python f-string, which uses {...} for substitution. ES6 template literals (${x}), object spread ({...obj}), and arrow functions conflict with this and cause build failures. Use string concatenation, function(){}, and Object.assign() instead.

Hard-Won Lessons

These are the bugs that cost hours. Keep them in mind while building.

Bubble Size Capping
Never cap radius with Math.min(r, maxR) — all large bubbles look identical. Use relative scaling: Math.sqrt(count/maxCount) * maxR.
Stacked Bars ≠ 100%
If a category exists in data but not in your order array, it silently gets excluded. Derive the array from data, don't hardcode.
Toggle Button Sync
When a toggle appears in multiple sections, the handler must update ALL button groups. Use i % 2 to reset index per pair.
Invisible Active State
White active background on a white page = invisible button. Use high-contrast: background:#002855; color:#fff.
5. Deploy

Push to GitHub Pages. One command, live in under two minutes.

git add index.html git commit -m "Update dashboard — $(date +%Y-%m-%d)" git push origin main

GitHub Pages config: Settings → Pages → Source: "Deploy from a branch" → Branch: main, folder: / (root).

Keep a single index.html at the repo root — this is the only file that matters. Archive old versions to an archive/ folder. Never have multiple HTML files in the root — it leads to confusion about which one is live.

If git push fails with an index.lock error, run rm -f .git/index.lock first. This happens when a previous git process crashed.

7. Adapting to Other Product Categories

This pipeline isn't specific to women's jeans. Here's what to change for a different category.

Phase 1 — New Category Pages
Navigate to the right category page on each retailer. The Instant Data Scraper process is identical.
Phase 2 — Update Selectors
CSS selectors change by category. Inspect the new product pages and update each scraper's selector config. The scraping architecture (crash recovery, concurrency, throttling) stays the same.
Phase 3 — New Classification Rules
Replace wash categories with whatever's relevant — e.g., for sneakers: "Running", "Casual", "Basketball". Update regex patterns accordingly.
Phase 4 — Adjust Dashboard Sections
Swap chart sections to match your data. Jeans have wash/fit/rise; sneakers might have cushioning/material/sport. The chart components (box plots, bubbles, stacked bars) are reusable.
The most time-consuming part of adapting is Phase 2 — finding the right selectors for each retailer's new category pages. Budget 30–60 minutes per retailer for selector discovery and testing. Everything else is relatively mechanical.
